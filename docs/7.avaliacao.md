# 7. Avaliação da Arquitetura

_Esta seção descreve a avaliação da arquitetura do Biblioo baseada no método ATAM (Architecture Tradeoff Analysis Method). Os cenários cobrem os principais atributos de qualidade definidos nos requisitos não funcionais e são sustentados por evidências quantitativas coletadas nos testes de performance executados com K6._

> **Nota sobre os dados:** todos os valores de latência medidos referem-se ao ambiente local de desenvolvimento (localhost). Para estimativa de produção aplica-se fator conservador de 4×, refletindo os recursos mais restritos do Google Cloud Run (1–2 Gi de memória, 1–2 vCPU) e dos serviços gerenciados utilizados (TiDB Cloud Serverless, Upstash, CloudAMQP Little Lemur, Neo4j Aura Free). Os thresholds dos RNFs foram definidos com esse fator já incorporado.

---

## 7.1. Cenários

**Cenário 1 — Desempenho sob carga normal sustentada:** o sistema deve manter latência dentro dos SLAs definidos nos RNFs enquanto serve múltiplos usuários simultaneamente realizando operações típicas do dia a dia (navegar no feed, buscar livros, consultar recomendações, interagir com o chat).

**Cenário 2 — Resiliência a picos abruptos de tráfego:** o sistema deve absorver picos repentinos de acesso (burst de 500–600 usuários simultâneos em questão de segundos) sem falhas funcionais, mesmo que com degradação controlada de latência, retornando a níveis normais após o pico.

**Cenário 3 — Segurança e controle de acesso:** o sistema deve garantir que dados privados de leitores só sejam acessíveis mediante autenticação válida, que conteúdo malicioso (XSS) não seja persistido e que endpoints críticos sejam protegidos contra uso abusivo.

**Cenário 4 — Disponibilidade com degradação graciosa:** quando serviços externos ou derivados ficarem indisponíveis (índice de busca, API de catálogo externo), o sistema deve continuar operacional usando fontes alternativas, sem retornar erros sistêmicos ao usuário.

**Cenário 5 — Escalabilidade do chat em tempo real:** o sistema deve suportar múltiplos leitores trocando mensagens simultâneas em comunidades distintas, entregando cada mensagem a todos os membros conectados com latência perceptível baixa.

### Cenário 2 — Resiliência a picos abruptos de tráfego

| | |
| --- | --- |
| **Atributo de Qualidade** | Desempenho / Confiabilidade |
| **Requisitos de Qualidade** | RNF-05, RNF-06, RNF-07, RNF-08, RNF-16, RNF-22 |
| **Preocupação** | Eventos externos (lançamento de funcionalidade, divulgação em redes sociais) podem gerar picos abruptos de acesso que excedam o volume esperado em uso normal, colocando o backend em risco de degradação severa ou falhas sistêmicas. |
| **Ambiente** | Sistema submetido a pico súbito de 500 VUs em 5 segundos (perfil `spike`), após base de 50 VUs. |
| **Estímulo** | Acesso massivo e repentino de usuários realizando operações mistas: login, leitura de feed, busca de livros, interação com comunidades e recomendações. |
| **Mecanismo** | Google Cloud Run escala automaticamente novas instâncias em segundos ao detectar concorrência acima do limite configurado (80 req/instância no portfolio, 200 no produção). O ambiente de produção mantém mínimo de 1 instância sempre ativa (sem cold start). Rate limiting via Bucket4j protege endpoints de maior impacto computacional. |
| **Medida de Resposta** | Latência p95 e taxa de falhas medidas nos testes `spike`. Nenhum domínio registrou falhas funcionais (5xx) nos picos testados. |

**Resultados dos testes `spike` (ambiente local, pico 500 VUs):**

| Feed social | 147 ms | ~588 ms | 0% | ✅ Atende RNF-06 |
| Publicar post | 352 ms | ~1 408 ms | 0% | ✅ Atende RNF-07 |
| Comentário no feed | 417 ms | ~1 668 ms | 0% | ✅ Dentro do aceitável |
| Review de livro | 463 ms | ~1 852 ms | 0% | ✅ Atende RNF-08 |
| 6 trilhas de recomendação | 1 280 ms | ~5 120 ms | 0% | ⚠️ Margem fina no RNF-16 |
| Roll Dice | 19 ms | ~76 ms | 0% | ✅ Excelente |
| Trending | 17 ms | ~68 ms | 0% | ✅ Excelente (cache Redis) |
| DNA Literário | 20 ms | ~80 ms | 0% | ✅ Excelente |

| Votação de livro | 521 ms | ~2 084 ms | 0% | ✅ Atende RNF-15 |

**Considerações:**

| **Riscos** | O endpoint de recomendações em pico (`spike`) atingiu p95 de 1 280 ms local, o que estima ~5 120 ms em produção — margem muito pequena em relação ao limite do RNF-16 de 5 000 ms. Um pico abrupto real em produção, combinado com cold start do portfolio, pode violar o SLA de recomendações. |
| --- | --- |
| **Pontos de Sensibilidade** | O portfolio (ambiente acadêmico) hiberna sem tráfego e pode sofrer cold start de 3–5 segundos na primeira requisição após hibernação. O ambiente de produção mantém instância sempre ativa, eliminando esse risco. |
| **Tradeoff** | A configuração de concorrência de 80 req/instância no portfolio (vs 200 no produção) escala mais rapidamente instâncias novas, mas cada instância nova tem custo de inicialização. No produção, 200 req/instância reduz o número de instâncias necessárias a custo de maior pressão por instância. |

### Cenário 3 — Segurança e controle de acesso

| | |
| --- | --- |
| **Atributo de Qualidade** | Segurança |
| **Requisito de Qualidade** | RNF-22 (taxa de erro < 1%) — associado também às restrições arquiteturais de autenticação e sanitização |
| **Preocupação** | Dados de leitura e publicações dos leitores são pessoais. O sistema deve garantir que apenas usuários autenticados acessem recursos protegidos, que conteúdo malicioso não seja armazenado e que endpoints sensíveis não sejam abusados. |
| **Ambiente** | Sistema em operação com usuários autenticados e tentativas de acesso não autorizado. |
| **Estímulo** | (a) Requisição a endpoint protegido sem token JWT válido. (b) Publicação de post com conteúdo contendo scripts maliciosos (XSS). (c) Leitor tenta acessar publicações de conta privada sem ser seguidor aprovado. (d) Chamadas excessivas ao assistente Bibo por um único usuário. |
| **Mecanismo** | (a) `JwtAuthenticationFilter` intercepta todas as requisições e rejeita com `401 Unauthorized` qualquer acesso sem token válido — endpoints públicos declarados explicitamente em `SecurityConfig`. (b) JSoup sanitiza todo conteúdo gerado por usuário antes da persistência, removendo tags e atributos não permitidos. (c) Regras de visibilidade verificadas na camada de domínio: perfis privados só expõem conteúdo a seguidores aprovados. (d) Rate limiting via Bucket4j limita o assistente a 20 requisições por minuto por usuário. |
| **Medida de Resposta** | (a) 100% das requisições sem token retornam `401` — nenhum endpoint protegido foi acessado sem autenticação nos testes. (b) Scripts injetados não aparecem no conteúdo persistido nem retornado. (c) Conteúdo de contas privadas inacessível a não seguidores — validado na suíte de testes de grafo social. (d) Requisições além do limite recebem `429 Too Many Requests`. |

**Considerações:**

| **Riscos** | Os tokens JWT mobile são armazenados em `FlutterSecureStorage`, protegidos pelo keystore do sistema operacional. Em dispositivos com root/jailbreak, essa proteção pode ser contornada — risco inerente a qualquer app mobile. |
| --- | --- |
| **Pontos de Sensibilidade** | O Refresh Token de 7 dias com rotação obrigatória exige que o cliente sempre armazene o token mais recente. Se o client perder o token rotacionado (crash durante troca), o usuário precisa re-autenticar. |
| **Tradeoff** | JWT stateless (sem revogação imediata) vs sessão com estado (revogação instantânea mas com custo de armazenamento e latência). A escolha por JWT stateless privilegia escalabilidade horizontal no Cloud Run, aceitando a janela de 1 hora de validade do access token mesmo após logout. |

### Cenário 4 — Disponibilidade com degradação graciosa

| | |
| --- | --- |
| **Atributo de Qualidade** | Disponibilidade / Confiabilidade |
| **Requisito de Qualidade** | RNF-22 (taxa de erro < 1%) |
| **Preocupação** | O sistema depende de serviços externos e derivados (OpenSearch, Google Books API, RabbitMQ) que podem ficar temporariamente indisponíveis. Uma falha nesses serviços não deve derrubar a plataforma nem retornar erros sistêmicos ao leitor. |
| **Ambiente** | Sistema em operação com um ou mais serviços derivados indisponíveis. |
| **Estímulo** | (a) OpenSearch indisponível durante busca de livros. (b) Google Books API com timeout ou erro durante busca de livro não catalogado. (c) Consumidor RabbitMQ recebe a mesma mensagem de evento mais de uma vez (redelivery). |
| **Mecanismo** | (a) Fallback automático para busca FULLTEXT no MySQL — o sistema retorna resultados do acervo interno sem retornar `5xx`. (b) O sistema retorna os livros já persistidos no MySQL e tenta novamente na próxima busca sem expor o erro ao leitor. (c) Todos os consumidores verificam o `event_id` antes de processar — mensagens duplicadas são descartadas silenciosamente (idempotência). |
| **Medida de Resposta** | Nos 72 testes K6 executados, nenhum domínio registrou falha funcional (`5xx`). A única taxa de erro relevante foi de 0,92% no `voting-load` — causada por race condition em fechamento concorrente de enquete (negócio), não por falha de infraestrutura. O Outbox Pattern garantiu zero mensagens perdidas mesmo sob carga de 600 VUs simultâneos. |

**Considerações:**

| **Riscos** | A race condition no fechamento de enquetes (`voting`) — onde múltiplos membros tentam encerrar a mesma votação simultaneamente — causa até 0,92% de falhas de negócio. A correção recomendada é lock otimista (`@Version`) na entidade de votação, ainda pendente no código. O mesmo padrão existe em join-requests de comunidade. |
| --- | --- |
| **Pontos de Sensibilidade** | O Bonsai.io Hobby (portfolio) tem limite de 125 MB de armazenamento para o índice OpenSearch. Quando o índice atinge esse limite, novos documentos não são indexados. Um serviço de limpeza semanal automático (`OpenSearchIndexCleanupService`) reconcilia o índice com o MySQL e remove documentos órfãos para manter o índice dentro do limite. |
| **Tradeoff** | O Outbox Pattern garante consistência entre persistência e mensageria, mas adiciona uma tabela de outbox e um worker de leitura que precisam ser monitorados. A alternativa (publicar diretamente no broker dentro da transação) é mais simples, mas cria risco de mensagens publicadas antes do commit ser confirmado em caso de falha. |

### Cenário 5 — Escalabilidade do chat em tempo real

| | |
| --- | --- |
| **Atributo de Qualidade** | Desempenho / Escalabilidade |
| **Requisitos de Qualidade** | RNF-12 (≤ 40 usuários simultâneos por comunidade sem perda), RNF-13 (entrega ≤ 5 000 ms) |
| **Preocupação** | O chat de comunidades usa WebSocket persistente — diferente de requisições HTTP comuns, conexões WebSocket mantidas por longo período consomem recursos de cada instância. Com múltiplas instâncias Cloud Run (escaladas automaticamente), mensagens de um cliente em uma instância precisam chegar a membros conectados em instâncias distintas. |
| **Ambiente** | Múltiplas comunidades ativas com membros conectados distribuídos entre instâncias Cloud Run. |
| **Estímulo** | 150–400 leitores conectados simultaneamente via WebSocket/STOMP em múltiplas comunidades, enviando e recebendo mensagens em tempo real. |
| **Mecanismo** | Cada instância Cloud Run mantém suas conexões WebSocket locais via SimpleBroker em memória. Quando uma mensagem é enviada, a instância publica em um RabbitMQ FanoutExchange. Todas as instâncias ativas são consumidoras desse exchange — cada uma entrega a mensagem aos seus membros conectados localmente. A afinidade de sessão (`--session-affinity`) no Cloud Run garante que o mesmo cliente sempre reconecte à mesma instância. |
| **Medida de Resposta** | Testes WebSocket/STOMP realizados com 160 VUs (load), 150 VUs (spike) e 250 VUs (stress): |

| Perfil | VUs WebSocket | Mensagens enviadas | Mensagens recebidas | Latência p95 entrega | Falhas | Integridade |
|---|---|---|---|---|---|---|
| load | 160 | 7 400 | 76 400 | 99 ms | 0% | 100% — sem duplicatas |
| spike | 150 | 12 810 | 351 400 | **11 ms** | 0% | 100% |
| stress | 250 | 15 145 | 309 000 | 27 ms | 0% | 100% |
| concurrency (400 VUs) | 400 | 30 790 | 4 900 000 | 449 ms | 0% HTTP | Sem duplicatas |

A entrega p95 de **11 ms** no spike confirma que o RNF-13 (≤ 5 000 ms) é amplamente atendido. O limite de 400 VUs WebSocket locais reflete a capacidade da máquina de desenvolvimento — não um bug de código: integridade de 100% foi confirmada em todos os cenários (zero duplicatas, zero mensagens perdidas).

**Considerações:**

| **Riscos** | O teste de concorrência com 400 VUs WebSocket simultâneos atingiu o limite de conexões locais da máquina de desenvolvimento — não do sistema em produção. Em produção, o Cloud Run escala automaticamente e o RabbitMQ FanoutExchange garante a entrega entre instâncias. |
| --- | --- |
| **Pontos de Sensibilidade** | A afinidade de sessão (`--session-affinity`) resolve a distribuição de conexões, mas em caso de redimensionamento forçado (instância removida pelo Cloud Run), clientes perdem a conexão e precisam reconectar. O app frontend reconecta automaticamente via sockjs-client, mas há uma janela de desconexão. |
| **Tradeoff** | O padrão FanoutExchange garante entrega a todos os membros de todas as instâncias, mas multiplica o volume de mensagens no broker: cada mensagem enviada por um usuário é consumida por todas as instâncias ativas, mesmo as que não têm membros da comunidade conectados. Com muitas instâncias ativas e poucas comunidades, isso gera tráfego desnecessário no broker. A alternativa (TopicExchange por comunidade) seria mais eficiente mas mais complexa de gerenciar. |

## 7.3. Avaliação Geral da Arquitetura

### Pontos Fortes

| Aspecto | Evidência |
|---|---|
| **Performance consistente** | 72/72 testes K6 aprovados; nenhuma falha funcional (5xx) em toda a suíte de carga; maior throughput de 1 538 req/s (user-stress, 600 VUs) |
| **Cache efetivo** | Trending e ShareCard com p95 < 40 ms mesmo sob 600 VUs — Redis como cache praticamente elimina variância de latência em endpoints de leitura intensiva |
| **Modularidade e isolamento** | Arquitetura hexagonal com 11 módulos independentes — alterações em um módulo não propagam efeitos colaterais aos demais; comunicação apenas via eventos (RabbitMQ) ou interfaces |
| **Resiliência a falhas externas** | Fallback automático do OpenSearch para MySQL FULLTEXT e da Google Books API para acervo local — zero `5xx` em simulações de falha de serviço derivado |
| **Idempotência garantida** | Todos os consumidores RabbitMQ verificam `event_id` — zero processamentos duplicados detectados nos testes de carga com até 600 VUs |
| **Chat escalável entre instâncias** | FanoutExchange + session affinity permite múltiplas instâncias Cloud Run com integridade de 100% na entrega de mensagens WebSocket |
| **Deploy sem downtime** | Cloud Run troca revisões de forma atômica — o pipeline CI/CD completo (commit → deploy em produção) opera em ~12 minutos |

### Limitações e Riscos Conhecidos

| Risco | Natureza | Ação Recomendada |
|---|---|---|
| Race condition em fechamento de votação | Bug real de concorrência — 0,92% de falhas no load com múltiplos VUs fechando a mesma enquete | Lock otimista (`@Version`) nas entidades `Voting` e `JoinRequest` |
| Recomendações próximas ao limite em pico | Estimativa de produção (p95 ~5 120 ms) excede marginalmente o RNF-16 (5 000 ms) em cenário de spike | Habilitar min-instances=1 no portfolio para eliminar cold start; avaliar cache de resultado de recomendações por usuário |
| Limite de armazenamento OpenSearch no portfolio | Bonsai.io Hobby tem 125 MB — crescimento do catálogo pode saturar o índice | Monitorar via serviço de limpeza semanal; migrar para plano pago se necessário |
| Dependência de free tiers em produção | Neo4j Aura Free, CloudAMQP Little Lemur e Bonsai.io Hobby não têm SLA garantido | Aceitável para estágio atual; planejar upgrade conforme crescimento de usuários |
| Janela de 1 hora do JWT | Logout não invalida o access token imediatamente — usuário desautenticado pode operar por até 1 hora com token antigo | Implementar lista negra de tokens em Redis (custo de latência) ou reduzir validade do access token |
