## 3.3. Restrições Arquiteturais

- O servidor deve ser desenvolvido exclusivamente em Java com Spring Boot. Nenhum outro framework ou linguagem de back-end é permitido.
- A persistência primária deve utilizar MySQL. O MySQL é a única fonte da verdade para todos os módulos; nenhum outro mecanismo de armazenamento (Redis, OpenSearch) pode assumir esse papel.
- Toda comunicação entre clientes e o back-end deve seguir o padrão RESTful sobre HTTP/HTTPS.
- O sistema deve autenticar usuários exclusivamente via tokens JWT, seguindo o modelo de Access Token (1 hora) e Refresh Token opaque com rotação obrigatória (7 dias). Todos os endpoints protegidos devem rejeitar requisições sem token válido com `401 Unauthorized`. Conexões WebSocket devem ser autenticadas no handshake via JWT.
- A interface web deve ser desenvolvida em Next.js e deve ser compatível exclusivamente com Google Chrome (v145+) e Safari (v26.3+).
- O aplicativo mobile deve ser desenvolvido em Flutter, com suporte a Android 10+ e iOS 16+ apenas.
- Toda comunicação assíncrona entre módulos internos (eventos de estante, feed, recomendações) deve trafegar exclusivamente pelo RabbitMQ. A comunicação síncrona direta entre módulos de domínio é proibida.
- O Redis deve ser utilizado como camada de cache (Cache-Aside) e não como fonte de verdade. Dados no Redis são sempre derivados e reconstruíveis a partir do MySQL.
- A busca de livros deve utilizar OpenSearch como índice derivado. O OpenSearch nunca é fonte primária; pode ser descartado e reconstruído integralmente a partir do MySQL. Em caso de falha, o sistema deve realizar fallback para busca FULLTEXT no próprio MySQL, sem retornar erro `5xx` ao cliente.
- A expansão do catálogo de livros deve utilizar a Google Books API como fonte de fallback. Livros retornados por essa API devem ser persistidos no MySQL imediatamente, de modo que buscas subsequentes sejam resolvidas internamente. Em caso de falha da API externa, o sistema deve retornar os livros já persistidos, sem retornar erro `5xx` ao cliente.
- Os módulos internos (Auth, Books, Shelf, Community, Feed, Identify, Recommendation) devem respeitar uma hierarquia de dependência estrita: módulos de nível inferior nunca importam nem conhecem módulos de nível superior. A comunicação entre módulos de domínio ocorre via eventos (RabbitMQ) ou via interfaces, nunca via chamadas HTTP entre serviços.
- Eventos de domínio devem ser gravados em tabela de outbox na mesma transação que a operação de negócio para consistência entre persistência e publicação de eventos. 
 

## 3.4. Mecanismos Arquiteturais

| **Análise** | **Design** | **Implementação** |
| --- | --- | --- |
| Persistência relacional | Mapeamento Objeto-Relacional (ORM) | Spring Data JPA + Hibernate + MySQL |
| Cache de dados | Cache-Aside (leitura com fallback ao banco) | Redis |
| Índice de busca | Índice invertido derivado e reconstruível | OpenSearch |
| Front-end web | SPA com renderização híbrida (SSR/CSR) | Next.js (React) |
| Front-end mobile | Framework cross-platform nativo compilado | Flutter (Dart) |
| Back-end | Arquitetura modular em camadas com API REST | Spring Boot (Java) |
| Comunicação assíncrona entre módulos | Mensageria com filas duráveis e ACK manual (Event-Driven Architecture + Outbox Pattern) | RabbitMQ |
| Comunicação em tempo real | Protocolo full-duplex orientado a eventos (push server → client) | WebSocket (Spring WebSocket) |
| Autenticação e autorização | Token stateless com versionamento + Refresh Token com rotação | Spring Security + JWT |
| Integração com sistema externo de catálogo | API REST com fallback e persistência local automática | Google Books API (HTTP/REST) |
| Geração de logs do sistema | Log estruturado por níveis com rastreamento de contexto | SLF4J + Logback (Spring Boot default) |
| Tratamento de exceções | Handler centralizado com mapeamento HTTP semântico | Spring `@ControllerAdvice` + RFC 7807 (Problem Details) |
| Teste de software | Testes unitários por camada + testes de integração com contexto Spring | JUnit 5 + Mockito + Spring Boot Test |
| Deploy | Containerização com orquestração de serviços | Docker + Docker Compose |
