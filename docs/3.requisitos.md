## 3.2. Requisitos Não-Funcionais

| **ID** | **Descrição** |
| --- | --- |
| RNF-01 | A interface web deve ser compatível com os navegadores Google Chrome (v145+) e Safari (v26.3+). |
| RNF-02 | A interface web deve oferecer layout responsivo para resoluções entre 1280px e 1920px de largura. |
| RNF-03 | A interface mobile deve oferecer layout responsivo para telas de 390px a 430px de largura. |
| RNF-04 | O sistema deve autenticar usuários via JWT. Todos os endpoints protegidos devem rejeitar requisições sem token válido. |
| RNF-05 | Conexões WebSocket devem ser autenticadas no handshake inicial via token JWT. Conexões sem token válido devem ser recusadas. |
| RNF-06 | O tempo de processamento de requisições à API deve ser inferior a 500ms, desconsiderando latência de rede. |
| RNF-07 | O endpoint de autocomplete de busca de livros deve responder em até 200ms em condições normais. |
| RNF-08 | O endpoint de listagem de comentários de uma comunidade deve responder em até 500ms no percentil p95. |
| RNF-09 | Livros retornados pela Google Books API devem ser persistidos no banco local. Buscas subsequentes pelo mesmo livro devem ser resolvidas internamente, sem nova chamada à API externa. |
| RNF-10 | O sistema deve suportar até 40 membros interagindo simultaneamente em uma comunidade de leitura, sem degradação perceptível no recebimento de comentários. |
| RNF-11 | A falha do OpenSearch não deve retornar erro `5xx` ao cliente. |
| RNF-12 | Em caso de falha do OpenSearch, o sistema deve realizar fallback para busca no banco de dados. |
| RNF-13 | A falha da Google Books API não deve retornar erro `5xx` ao cliente. |
| RNF-14 | Em caso de falha da Google Books API, o sistema deve retornar os livros já persistidos. |
| RNF-15 | O aplicativo mobile deve ser compatível com Android 10+ e iOS 16+. |
| RNF-16 | O feed social deve carregar e exibir os primeiros itens em até 10 segundos após o login do usuário. |
| RNF-17 | A página de perfil do usuário deve carregar em até 5 segundos em acessos repetidos. |
| RNF-18 | A página de uma comunidade de leitura deve carregar em até 5 segundos em acessos repetidos. |

## 3.3. Restrições Arquiteturais

_Enumere as restrições arquiteturais. Lembre-se de que as restrições arquiteturais geralmente não são consideradas requisitos uma vez que limitam a solução candidata. Os requisitos não impõem restrição, mas precisam ser satisfeitos._

As restrições impostas ao projeto que afetam sua arquitetura são (por exemplo):

- O software deverá ser desenvolvido em Python/Django;
- A comunicação da API deve seguir o padrão RESTful.

## 3.4. Mecanismos Arquiteturais

_Visão geral dos mecanismos que compõem a arquitetura do sosftware baseando-se em três estados: (1) análise, (2) design e (3) implementação. Em termos de Análise devem ser listados os aspectos gerais que compõem a arquitetura do software como: persistência, integração com sistemas legados, geração de logs do sistema, ambiente de front end, tratamento de exceções, formato dos testes, formato de distribuição/implantação (deploy), entre outros. Em Design deve-se identificar o padrão tecnológico a seguir para cada mecanismo identificado na análise. Em Implementação, deve-se identificar o produto a ser utilizado na solução.
 Ex: Análise (Persistência), Design (ORM), Implementação (Hibernate)._
