## 3.3. Restrições Arquiteturais

**Back-end**

- O servidor deve ser desenvolvido exclusivamente em Java 25 com Spring Boot 4. Nenhum outro framework ou linguagem de back-end é permitido.
- A arquitetura do back-end deve seguir o estilo **Hexagonal (Ports & Adapters)** em um **monólito modular** com 11 módulos de domínio independentes: `user`, `books`, `community`, `dna`, `feed`, `notification`, `recommendation`, `share`, `trending`, `assistant` e `infrastructure`. Módulos de domínio nunca importam nem conhecem outros módulos de domínio; a comunicação entre eles ocorre exclusivamente via eventos assíncronos (RabbitMQ) ou via interfaces — nunca por chamada direta.
- A persistência relacional primária deve utilizar MySQL (desenvolvimento local) e TiDB Cloud Serverless (produção). O banco relacional é a única fonte da verdade; nenhum outro mecanismo de armazenamento pode assumir esse papel.
- O grafo de relacionamentos entre leitores e livros para as trilhas de recomendação deve utilizar Neo4j (desenvolvimento local) e Neo4j Aura Free (produção), acessado exclusivamente via driver Java nativo com consultas Cypher diretas. O uso de Spring Data Neo4j é proibido para preservar controle total sobre as consultas de recomendação.
- O Redis (desenvolvimento local) e Upstash (produção) devem ser utilizados exclusivamente como camada de cache (Cache-Aside) e para armazenamento de estado efêmero (histórico do assistente, parâmetros do algoritmo de recomendação). Dados no Redis são sempre derivados e reconstruíveis a partir do MySQL. O cache nunca deve armazenar valores nulos para prevenir cache poisoning.
- A busca de livros e usuários deve utilizar OpenSearch como índice derivado e reconstruível a partir do MySQL. Em produção, o ambiente portfolio usa Bonsai.io Hobby (HTTPS, gratuito permanente) e o ambiente de produção usa uma instância própria em Google Compute Engine via rede interna VPC. Em caso de falha do OpenSearch, o sistema deve realizar fallback para busca FULLTEXT no MySQL, sem retornar erro `5xx` ao cliente.
- O catálogo de livros pode ser expandido via Google Books API como fonte de fallback. Livros retornados por essa API devem ser persistidos no MySQL imediatamente para que buscas subsequentes sejam resolvidas internamente. Em caso de falha da API externa, o sistema deve retornar os livros já persistidos, sem retornar erro `5xx` ao cliente.
- Toda comunicação assíncrona entre módulos (eventos de estante, feed, recomendações, notificações) deve trafegar exclusivamente pelo RabbitMQ (desenvolvimento local) e CloudAMQP Little Lemur (produção). Todos os consumidores RabbitMQ devem verificar o `event_id` antes de processar para garantir idempotência.
- Eventos de domínio devem ser gravados em tabela de outbox na mesma transação da operação de negócio, e só então publicados no broker. Isso garante consistência entre a persistência da entidade e a publicação do evento.
- O sistema deve gerenciar sessões exclusivamente via tokens JWT, com Access Token de 1 hora e Refresh Token de 7 dias com rotação obrigatória. A autenticação pode ser realizada por e-mail e senha ou via conta Google (OAuth 2.0); em ambos os casos o sistema emite internamente o par de tokens JWT. Todos os endpoints protegidos devem rejeitar requisições sem token válido com `401 Unauthorized`. Conexões WebSocket devem ser autenticadas no handshake via JWT antes de qualquer mensagem ser aceita.
- Todo conteúdo gerado por usuários (posts, reviews, comentários, bio, nome de comunidade) deve ser sanitizado contra XSS antes da persistência. Arquivos enviados pelo usuário (avatares, banners, imagens e GIFs de posts, de comentários e de comunidades) devem ter seu tipo real detectado antes do processamento, com validação de tamanho máximo, e ser armazenados de forma assíncrona no Cloudinary, serviço externo gerenciado de hospedagem e entrega de mídia.
- O código back-end deve seguir o Google Java Format, aplicado obrigatoriamente via Spotless antes de qualquer commit. Comentários, logs e mensagens de exceção devem ser escritos em português do Brasil; o código-fonte em inglês.

**Front-end web**

- A interface web deve ser desenvolvida em Next.js 16 com App Router e React 19, utilizando TypeScript estrito sem `any` implícito.
- A interface web deve ser compatível exclusivamente com Google Chrome (v145+) e Safari (v26.3+).
- Todas as chamadas HTTP à API devem passar pela camada de serviços (`src/services/`). Componentes de domínio nunca chamam a API diretamente.
- A composição de classes CSS deve utilizar o utilitário `cn()` (clsx + tailwind-merge) obrigatoriamente para prevenir conflitos de classes Tailwind.

**Aplicativo mobile**

- O aplicativo mobile deve ser desenvolvido em Flutter 3.11+ com suporte exclusivo a Android 10+ e iOS 16+.
- O app deve funcionar em modo **offline-first**: exibir dados do cache local (Drift/SQLite e Hive) sem conexão e sincronizar com a API quando a rede retornar. O Repository deve sempre tentar o datasource local antes de consultar o remoto.
- Tokens JWT devem ser armazenados exclusivamente em `FlutterSecureStorage` (armazenamento criptografado pelo sistema operacional). É proibido armazená-los em SharedPreferences ou qualquer memória não criptografada.
- A lógica de negócio deve ser separada da UI pelo padrão BLoC: BLoCs chamam apenas Repositories; Screens/Widgets reagem via BlocBuilder e BlocListener; módulos de feature não importam outros módulos de feature.

**Deploy e infraestrutura**

- O back-end deve ser empacotado como imagem Docker e publicado no Google Artifact Registry a cada build. O deploy deve ser realizado em dois ambientes independentes no Google Cloud Run (us-central1): `biblioo-portfolio` (0–2 instâncias, 1 Gi/1 vCPU, hiberna sem tráfego) e `biblioo-producao` (1–10 instâncias, 2 Gi/2 vCPU, sempre ativa).
- Todos os secrets de produção (credenciais de banco, chaves de API, tokens) devem ser armazenados exclusivamente no Google Secret Manager e injetados nos serviços via `--set-secrets`. Nenhum secret pode ser exposto em código-fonte, variáveis de ambiente em texto claro ou logs.
- O pipeline de CI/CD deve ser acionado automaticamente por push na branch `prod`, espelhada do repositório privado para o público via GitHub Actions, com build e deploy executados pelo Cloud Build em aproximadamente 12 minutos sem downtime.
- O Cloud Run deve ser configurado com `--session-affinity` para garantir que conexões WebSocket/STOMP de um mesmo cliente permaneçam na mesma instância. Mensagens entre instâncias devem ser roteadas via RabbitMQ FanoutExchange.
